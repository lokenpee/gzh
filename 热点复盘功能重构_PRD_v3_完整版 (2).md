# 热点复盘功能重构 PRD v3（完整版）

## 一、产品定位

热点复盘是一个面向领导层的热点事件传播全景面板。用户输入一个事件关键词和时间区间，系统自动从公司数据库中检索相关公众号文章，以可视化方式回答三个问题：**这件事哪天讨论最多？最火的文在说什么？涉及哪些票？**

不做文章质量评分，不做公众号排名，不做溯源预测。只做客观统计和直观展示。

---

## 二、系统架构与数据流

```
用户输入（时间区间 + 热点关键词）
→ 意图识别（LLM + 联网搜索）→ event_name, event_entities, key_points, Industry_asset, search_window
→ 数据提取（查询公司数据库 CJ_NEWS / CJ_NEWS_CONTENT / CJ_NEWS_OPEN）→ 文章列表
→ UTS推送 → 本地后台
  → Embedding 粗筛（语义相似度过滤，保留最多300篇）
  → LLM 精筛（篇级相关性判定，输出最终文章列表）
  → 各模块聚合计算
→ UI展示（传播时间轴 / 当日热点文章榜 / 观点总结 / 热点文章卡片墙 / 关联标的走势）
```

## 三、数据层：可用字段清单

以下字段全部来自公司数据库，通过UTS接口推送至本地后台。

### 3.1 公众号文章主表（CJ_NEWS）

| 字段名 | 字段类型 | 含义 | 模块用途 |
|--------|----------|------|----------|
| ID | NUMBER(38) | 文章唯一标识 | 关联CJ_NEWS_CONTENT |
| TITLE | VARCHAR2(360) | 文章标题 | 全部展示模块 |
| URL | VARCHAR2(600) | 原文链接 | 点击跳转 |
| DTIME | DATE | 文章发布时间（精确到秒） | 时间轴分组、文章排序 |
| SOURCE | VARCHAR2(50) | 公众号名称 | 文章榜、卡片墙、观点总结 |
| AUTHOR | VARCHAR2(50) | 文章作者 | 文章榜展示 |
| READ_COUNT | INTEGER | 阅读量，默认0 | 文章排序（降序）、卡片墙展示 |
| 狗SUMMARY | VARCHAR2(2000) | 核心摘要（算法识别） | 摘要展示，优先使用 |
| ABSTRACT_TEXT | VARCHAR2(500) | 资讯摘要 | SUMMARY为空时降级使用 |
| KEYWORDS | VARCHAR2(200) | 自动提取的关键词列表 | 辅助判断文章相关性 |
| 狗SCODE_LIST | VARCHAR2(150) | 关联股票代码（逗号分隔） | 关联标的模块 |
| SNAME_LIST | VARCHAR2(150) | 关联股票名称（逗号分隔） | 关联标的模块 |
| CJ_TYPE | INTEGER | 2=原创/1=资讯聚合/3=期货原创 | 筛选原创（仅保留CJ_TYPE=2） |
| ATTACH_FILE2 | VARCHAR2(200) | 首图附件名 | 卡片墙封面图（拼接图片服务地址） |

### 3.2 文章内容表（CJ_NEWS_CONTENT）

| 字段名 | 字段类型 | 含义 | 模块用途 |
|--------|----------|------|----------|
| NEWS_ID | INTEGER | 关联CJ_NEWS.ID | JOIN条件 |
| CONTENT | CLOB | 文章完整正文 | 数据提取时的关键词匹配源 |

### 3.3 公众号信息表（CJ_NEWS_OPEN）

| 字段名 | 字段类型 | 含义 | 模块用途 |
|--------|----------|------|----------|
| ID | INTEGER | OPENID | 关联CJ_NEWS.OPENID |
| NAME | VARCHAR2(100) | 公众号名称 | 展示用（CJ_NEWS.SOURCE为空时降级使用） |
| DESCRIPTION | VARCHAR2(500) | 公众号简介 | 备用 |

### 3.4 数据提取的JOIN逻辑

CJ_NEWS 左关联 CJ_NEWS_CONTENT（ON CJ_NEWS.ID = CJ_NEWS_CONTENT.NEWS_ID），左关联 CJ_NEWS_OPEN（ON CJ_NEWS.OPENID = CJ_NEWS_OPEN.ID）。查询条件为：DTIME落在search_window范围内，且CJ_TYPE = 2（原创），且CJ_NEWS_CONTENT.CONTENT同时包含所有event_entities（AND逻辑）。

## 四、核心处理引擎

### 4.1 用户输入

| 输入项 | 必填 | 类型 | 说明 |
|--------|------|------|------|
| 时间区间 | 是 | Date Range | 格式 yyyy-MM-dd |
| 热点事件 | 是 | Text | 用户输入的事件关键词，如"海湖庄园协议" |

### 4.2 意图识别

调用大模型API + 联网搜索工具，解析用户输入的事件关键词。

**输入：**
- event_keyword（用户输入的原始关键词）
- time_range（用户选择的时间区间start和end）

**输出结构：**

```
event_name: "规范化事件名，不超过30字，包含核心主体+关键动作+时间/地点（如有）"
event_entities: ["实体A", "实体B", "实体C"]  // 2-3个核心实体词，用于数据库关键词AND匹配
key_points: [
  {"node_name": "节点描述", "time": "yyyy-MM-dd"},
  ...  // 最多5个，改变事件走向的关键节点
]
Industry_asset: ["标的名称1", "标的名称2"]  // 事件关联的板块或标的，用于关联标的走势模块
search_window: {"start": "yyyy-MM-dd", "end": "yyyy-MM-dd"}  // 等于用户输入的时间区间
```

**纪律：**
- event_name必须基于联网搜索结果生成，严禁依赖模型记忆推断。
- event_entities必须2-3个，按重要性排序，禁止编造未在搜索结果中出现的实体。
- key_points提取事件发展中改变走向的关键节点，最多5个，每个含描述和时间。
- Industry_asset为该事件关联的A股板块或个股名称列表，如["人形机器人","黄金"]。
- 输出必须是合法JSON，不含Markdown代码块标记。

### 4.3 数据提取

使用意图识别产出的event_entities作为核心实体词，向公司数据库发起查询。

**查询逻辑：**
- DTIME >= search_window.start 且 DTIME <= search_window.end + 23:59:59
- CJ_TYPE = 2（仅原创）
- CJ_NEWS_CONTENT.CONTENT 同时包含全部 event_entities（AND逻辑）

**零召回降级：**
- 若3个实体AND匹配无结果 → 降级为"任意2个AND" → 仍无则"任意1个OR"
- 所有降级策略用尽仍无结果 → 返回空列表，UI进入全局空状态

**输出字段：**

将满足条件的文章以列表形式推送至本地后台，每条记录包含以下字段：

| 字段名 | 来源表 | 来源字段 | 说明 |
|--------|--------|----------|------|
| article_id | CJ_NEWS | ID | 文章唯一ID |
| title | CJ_NEWS | TITLE | 标题 |
| url | CJ_NEWS | URL | 原文链接 |
| pub_datetime | CJ_NEWS | DTIME | 发布时间（含时分秒） |
| source_name | CJ_NEWS | SOURCE | 公众号名称，为空则降级取CJ_NEWS_OPEN.NAME |
| author | CJ_NEWS | AUTHOR | 作者 |
| read_count | CJ_NEWS | READ_COUNT | 阅读量 |
| summary | CJ_NEWS | SUMMARY | 优先取SUMMARY，为空则降级取ABSTRACT_TEXT |
| keywords | CJ_NEWS | KEYWORDS | 关键词列表 |
| stock_codes | CJ_NEWS | SCODE_LIST | 关联股票代码 |
| stock_names | CJ_NEWS | SNAME_LIST | 关联股票名称 |
| cover_image | CJ_NEWS | ATTACH_FILE2 | 首图附件名，需拼接图片服务前缀获取URL |

### 4.4 本地后台处理

UTS 推送文章列表至本地后台后，依次经过 Embedding 粗筛和 LLM 精筛两轮过滤，再进行聚合计算，供各展示模块消费。

**整体流程**：

```
UTS推送原始文章列表
→ Embedding 粗筛（语义过滤，剔除关键词命中但主题无关的文章）→ 最多300篇
→ LLM 精筛（逐篇判定是否实质性讨论该事件）→ 最终文章列表
→ 各模块聚合计算
```

---

#### **Embedding 粗筛**

用余弦相似度对 UTS 推送的文章做语义层过滤。



- **Query 构建**：

  - 文本：`{event_name}。`
  - 前缀：`为这个句子生成表示以用于检索相关文章：{query}`

  **Document 构建**：

  - 文本：`{TITLE}。{SUMMARY}`，SUMMARY 为空则降级取 ABSTRACT_TEXT，再空取 CONTENT 前 200 字
  - 总长度 ≤ 256 tokens（约 400 字符），超长的优先截断摘要，保留标题完整
  - 不用全文：标题+摘要覆盖了 85%+ 有效语义，全文信噪比低、推理成本高、边际效益 < 5%
  - 前缀：`为这个段落生成表示：{doc}`

  **相似度计算**：余弦相似度 `score = cos_sim(query_emb, doc_emb)`，实际分布通常在 0.2-0.85。

  **截断策略（自适应分位数 + 硬边界）**：

  ```
  threshold = max(P75_score, 0.50)
  threshold = min(threshold, 0.75)
  保留 score >= threshold 的文章
  硬边界修正：
    - 若结果 > 300篇 → 取 Top 300
    - 若结果 < 10篇 → 取 Top 10（强制保底）
    - 若关键词 AND 召回总数 <= 20篇 → 跳过 Embedding，全部进入 LLM 精筛
  ```

  | 参数        | 取值         | 依据                                                         |
  | ----------- | ------------ | ------------------------------------------------------------ |
  | P75 分位数  | 第 75 百分位 | 粗筛定位是"剔除尾部无关"，P75 保留前 25%，足够宽松又有效截断噪音 |
  | 下限 0.50   | 固定值       | BGE-large-zh MTEB 经验断点，低于 0.50 语义夹角 > 60 度，通常"提及关键词但主题偏离" |
  | 上限 0.75   | 固定值       | 防止语义高度集中事件整体 score 都高，导致阈值过高、召回不足  |
  | Top 300     | 硬上限       | LLM 精筛每批 20 篇 × 约 15 批，5 批并发约 6-8 秒，超过则成本和时间不可控 |
  | Top 10 保底 | 硬下限       | 下游"观点总结"需 Top10、"卡片墙"需 Top3，至少保 10 篇        |
  | ≤20 篇跳过  | 条件触发     | 小样本下 embedding 区分度有限（P75 与 P25 可能只差 0.05），不如直接交 LLM |

  **兜底机制**：

  | 异常场景                | 触发条件                 | 处理策略                                              |
  | ----------------------- | ------------------------ | ----------------------------------------------------- |
  | Embedding 服务宕机/超时 | 3 秒内未返回             | **跳过 Embedding**，全部文章直接进入 LLM 精筛         |
  | 所有 score < 0.35       | 分布整体偏低             | 取 Top 50 进 LLM，不再用阈值过滤                      |
  | 向量维度异常            | 输出维度 ≠ 1024          | 丢弃异常向量继续计算；异常比例 > 30% 则跳过 Embedding |
  | 单篇文章字段全空        | 无 title/summary/content | 用 keywords + title 拼接；keywords 也为空则 score = 0 |

**LLM 精筛**

对 Embedding 粗筛后的文章逐篇判定是否实质性讨论该事件。

**每批 20 篇、并发 5 批的依据**：超过 20 篇/批准确率下降 8-12%（LLM 注意力分散）；低于 15 篇/批 API 调用次数过多、总耗时长；5 批并发不触发 GPT-4o-mini 的 TPM 限流，同时 100 篇/轮可在 2-3 轮处理完 200 篇。

**Prompt 设计**：

```
System：你是一名财经内容分析专家。判断每篇文章是否与热点事件"实质性相关"。
       【相关】围绕事件展开实质性讨论（分析、解读、评论、影响评估等）
       【不相关】仅提及关键词但主题无关；事件只是边缘信息（一句话带过）
       只基于标题和摘要判断，严禁猜测正文。
       输出 JSON 数组，字段：article_id、related（布尔值）。不要输出解释。

User：热点事件：{event_name}
      核心实体：{entities.join("、")}
      事件关键节点：{key_points.map(k => k.node_name).join("；")}

      [1] ID:{id} 标题：{title} 摘要：{summary}
      [2] ID:{id} 标题：{title} 摘要：{summary}
      ...（最多 20 篇）
```

给 LLM 提供 `event_name` + `entities` + `key_points` 三重上下文：event_name 锚定事件整体语义，entities 防同名异义，key_points 帮助识别事件不同侧面（政策面/市场面/国际影响面），避免 LLM 只认某一类文章。

max_tokens=2000（20 篇 × 30 tokens 输出 = 600，留足余量防截断）。单批超时 10 秒（4o-mini 通常 2-4 秒返回）。

**结果处理**：合并所有批次的 `article_id → related` 映射，去重后保留 `related == true` 的文章。

**兜底机制**：

| 异常场景        | 触发条件          | 处理策略                                                     |
| --------------- | ----------------- | ------------------------------------------------------------ |
| API 超时/报错   | 10 秒未返回或 5xx | 重试 2 次（间隔 2s→4s 指数退避），仍失败则**该批全部保留**（related=true），记录日志 |
| JSON 格式不合法 | 无法解析为数组    | 正则尽力提取 `article_id` 和 `related`；仍失败则该批全部保留 |
| 某批全判 false  | 20 篇全不相关     | **二次确认**：用含正例的 prompt 重调一次；仍全 false 则取该批 Embedding Top 50% 保留 |
| 精筛后 < 5 篇   | 合并后 ≤ 5 篇     | **自动补回**：从 Embedding 结果按 score 取 Top 10 补回       |
| 精筛后 = 0 篇   | 全部 false        | 取 Embedding Top 20 作为最终结果，UI 提示"已展示语义最接近的文章" |
| 429 Rate Limit  | TPM 超限          | 等待 60s 重试，并发降为 2 批                                 |

---

**聚合计算（精筛后的文章列表）**：

**时间轴聚合：** 将文章列表按pub_datetime的日期部分（yyyy-MM-dd）分组，每组计算article_count = COUNT(*)。时间轴横轴从search_window.start到search_window.end连续覆盖，无文章日期记为0。

**阅读量排序：** 文章列表按read_count降序排列，read_count相同时按pub_datetime降序排列，作为"当日文章榜"和"观点总结"的排序依据。

**Top10选取：** 整个时间区间内read_count最高的10篇文章，作为"观点总结"模块的输入。

**Top3选取：** 整个时间区间内read_count最高的3篇文章，作为"热点文章卡片墙"的输入。

## 五、展示层

### 5.1 传播时间轴

**目标：** 展示选定时间区间内，每天与该事件相关的原创文章数量折线，直观反映事件发酵程度——哪天爆了、哪天退了、整体趋势如何。

**图表类型：** ECharts 平滑面积折线图。

**布局：** 页面顶部，通栏宽度，高度360px。标题"事件传播趋势"位于左上角。

**数据输入：**

| 字段 | 来源 | 处理方式 |
|------|------|----------|
| date | pub_datetime截断日期 | 按日分组，生成连续日期序列 |
| count | 后台聚合计算 | 当日文章数量，无数据日期补0 |
| search_window | 意图识别输出 | 定义横轴起始和结束边界 |
| key_points | 意图识别输出 | 在时间轴上标注事件关键节点 |

**视觉规范：**
- 折线颜色 #2563eb，线宽2.5px，平滑曲线（smooth: 0.4）。
- 面积渐变填充，从 rgba(37,99,235,0.15) 过渡到 rgba(37,99,235,0.02)。
- X轴为日期轴，刻度格式MM-DD，标签颜色 #6b7280，轴线颜色 #e5e7eb。密集时自动旋转45度防重叠。
- Y轴为文章数量，整数刻度，最小值0，最大值自适应（向上取整到最近5的倍数），网格线 #f3f4f6 虚线。
- 每个数据点顶部显示数量标签（仅count > 0时）。
- 横轴支持缩放（dataZoom），默认展示完整区间。

**关键节点标注：**
- 来源：key_points数组（最多5个），每项含node_name和time。
- 展示：金色虚线竖线（颜色 #d4a855，线宽1.5px，虚线模式[6, 4]）贯穿图表区域。
- 虚线顶部外侧标注node_name（字体11px，颜色 #92400e，背景 rgba(251,191,36,0.15)，圆角4px），底部标注日期（格式MM-DD，字体10px，颜色 #d4a855）。
- 多个节点标签上下错开展示（间距20px）。

**交互设计：**
- 点击count > 0的数据点 → 触发"当日热点文章榜"模块（5.2）加载该日文章列表。
- 被点击的数据点高亮显示（圆点放大1.5倍 + 外发光）。
- Tooltip悬停：背景白色，圆角8px，阴影 0 4px 12px rgba(0,0,0,0.08)，内部显示日期和文章数量。

**空状态：** 若时间轴所有日期count均为0，正常渲染空白折线图，图表区域中央叠加文字"当前时间区间内暂无相关文章"（颜色 #9ca3af，14px）。

### 5.2 当日热点文章榜

**目标：** 点击时间轴上某一天后，展示该日期（00:00-23:59）内所有文章的阅读量排行榜。一眼看到"4月2号这天最火的文章是哪些"。

**触发条件：** 用户在"传播时间轴"模块中点击了某个count > 0的数据点。

**布局：** 位于传播时间轴下方，独立卡片，圆角12px，内边距20px。标题为"YYYY-MM-DD 当日热文"（日期动态替换），右侧显示该日文章总数。

**数据过滤：** 从文章列表中筛选pub_datetime日期部分等于被点击日期的记录，按read_count降序排列。

**表格字段：**

| 列名 | 字段来源 | 列宽 | 对齐 | 说明 |
|------|----------|------|------|------|
| 排名 | 本地序号（1,2,3...） | 50px | 居中 | 前三名分别用金色/银色/铜色圆点标识 |
| 标题 | title | 自适应（占剩余宽度60%） | 左对齐 | 单行截断，hover显示完整标题Tooltip |
| 来源 | source_name | 自适应（占剩余宽度20%） | 左对齐 | 公众号名称 |
| 作者 | author | 120px | 左对齐 | 为空则显示"-" |
| 阅读量 | read_count | 100px | 右对齐 | 千分位格式化，如"12,580" |
| 时间 | pub_datetime | 90px | 右对齐 | 格式 HH:mm |

**交互设计：**
- 点击标题 → 新标签页打开url。
- 表格行hover → 背景色变为 #f8fafc。
- 最多展示前20条，超出时表格内滚动。
- 底部显示"共N篇"统计提示。

**空状态：** 若当日无文章（理论上不应发生，因时间轴只显示有数据日期），展示"当日暂无文章"。

### 5.3 观点总结

**目标：** 对整个时间区间内所有文章，按阅读量取Top10，用大模型提炼每篇文章的核心观点。看到的是"这段时间最火的10篇文章，各自说了什么判断"。

**处理范围：** 整个search_window内read_count最高的10篇文章。

**大模型调用逻辑：**

将这10篇文章的title和summary（summary为空则取content前500字）一次性输入大模型，要求对每篇文章提炼核心观点。

**大模型输出结构（每篇）：**

```
{
  "viewpoint": "一句话核心观点，不超过30字",
  "stance": "看多/看空/中性/风险预警"  // 基于文章明确表述判定，无明显立场标"中性"
}
```

**大模型纪律：**
- 严禁编造观点，必须基于文章内容提炼。
- 立场判定必须基于文章明确表述，不得推测。
- 单次调用完成10篇文章的提炼。
- 输出为合法JSON数组。

**布局：** 独立卡片，圆角12px，内边距20px。标题"热门观点提炼"，右侧标注"基于阅读量Top10"。

**表格字段：**

| 列名 | 字段来源 | 列宽 | 对齐 | 说明 |
|------|----------|------|------|------|
| 排名 | 序号1-10 | 40px | 居中 | 数字排名 |
| 观点 | LLM输出 viewpoint | 280px | 左对齐 | 核心观点摘要，最多2行 |
| 立场 | LLM输出 stance | 80px | 居中 | Tag组件：看多(绿色)/看空(红色)/风险预警(橙色)/中性(灰色) |
| 文章标题 | title | 自适应 | 左对齐 | 单行截断，点击跳转url |
| 来源 | source_name | 140px | 左对齐 | 公众号名称 |
| 阅读量 | read_count | 90px | 右对齐 | 千分位格式化 |

**交互设计：**
- 点击文章标题 → 新标签页打开url。
- 表格行hover → 行背景变色，标题加下划线。
- 按read_count降序排列（与排名列一致）。

**空状态与降级：**
- 若文章总数为0 → 隐藏本模块。
- 若文章数 < 10 → 按实际数量提炼和展示。
- 若大模型调用失败 → 展示文章基本信息（标题、来源、阅读量），观点列和立场列显示"提炼失败"（灰色文字）。

### 5.4 热点文章卡片墙

**目标：** 以卡片形式展示区间内阅读量最高的3篇文章，每篇卡片包含封面图、标题、简短摘要、来源和发布时间。直观呈现热点内容的视觉全貌。

**数据范围：** 整个search_window内read_count最高的3篇文章。

**布局：** 位于"传播时间轴"和"观点总结"之间（或页面适当位置）。横向排列3张卡片，等宽分布，间距16px。卡片圆角12px，背景白色，边框1px solid #e5e7eb，阴影 0 2px 8px rgba(0,0,0,0.04)。

**单张卡片结构（从上到下）：**

**封面图区域：**
- 来源：cover_image（CJ_NEWS.ATTACH_FILE2）拼接图片服务地址获取封面图URL。
- 高度160px，宽度100%，object-fit: cover，顶部圆角12px。
- 若cover_image为空 → 展示渐变色占位背景（颜色基于文章ID哈希生成，确保同一文章颜色稳定）。
- 右上角叠加阅读量Badge：背景 rgba(0,0,0,0.6)，白色文字，圆角4px，格式"12.5万阅读"或"1,234阅读"。

**内容区域：**
- 内边距16px。

| 元素 | 字段来源 | 样式说明 |
|------|----------|----------|
| 标题 | title | 16px，font-weight 600，颜色 #111827，最多2行，超行截断 |
| 摘要 | summary（SUMMARY降级ABSTRACT_TEXT） | 14px，颜色 #6b7280，最多2行，超行截断 |
| 来源 | source_name | 12px，颜色 #9ca3af，格式"来源：{公众号名}" |
| 发布时间 | pub_datetime | 12px，颜色 #9ca3af，格式"MM-DD HH:mm" |

**交互设计：**
- 整张卡片可点击 → 新标签页打开url。
- Hover → 卡片上浮（translateY -4px），阴影加深至 0 8px 24px rgba(0,0,0,0.08），过渡200ms。
- 3张卡片横向排列，卡片容器支持横向滑动（移动端适配）。

**空状态：** 若文章不足3篇，按实际数量展示，剩余位置留空。若文章数为0，隐藏本模块。

### 5.5 关联标的走势

**目标：** 展示事件关联标的在选定时间区间内的日K线走势，并在K线上标注事件关键节点，方便对照"事件发生 vs 市场反应"。

**当前状态说明：** 由于行情数据源尚未接入，本模块暂使用模拟数据展示。模拟数据需保证同一标的在同一时间范围内生成的数据稳定一致。

**标的来源：** 意图识别产出的Industry_asset数组，如["人形机器人", "黄金", "韩国综指"]。

**布局：** 独立卡片，圆角12px，内边距20px。最小宽度800px。顶部标题区左侧"关联标的走势"，右侧为标的切换Tab。

**标的切换Tab：**
- 数据来源：Industry_asset数组。
- 样式：分段控制器风格，Tab项数量为数组长度，文案为标的名称。
- 点击切换标的，触发K线图重新渲染。
- Industry_asset为空时，隐藏整个模块。

**K线图展示：**

| 字段 | 来源 | 说明 |
|------|------|------|
| date | 模拟生成 | 交易日日期，yyyy-MM-dd |
| open | 模拟生成 | 开盘价 |
| close | 模拟生成 | 收盘价 |
| high | 模拟生成 | 最高价 |
| low | 模拟生成 | 最低价 |
| volume | 模拟生成 | 成交量（预留，暂不展示） |

**模拟数据生成规则：**
- 基于标的名称 + 时间范围起始日期作为随机种子，使用伪随机数生成算法产生模拟K线数据。
- 同一标的在同一时间范围内每次生成的数据完全一致。
- 数据形态需合理：收盘价在合理区间波动（A股个股10-300元，指数1000-10000点），日涨跌幅度控制在±10%以内。
- 只生成search_window范围内的交易日数据（排除周末和法定节假日）。
- K线为蜡烛图：收盘 >= 开盘 → 红色 #ef4444；收盘 < 开盘 → 绿色 #22c55e。红涨绿跌。
- 高度400px，宽度自适应。Y轴左侧价格刻度，X轴底部日期刻度。支持dataZoom缩放。

**事件关键节点标注：**
- 数据来源：key_points（意图识别输出），每项含node_name和time。
- 展示：金色虚线竖线（dashed，线宽1.5px，线型[6, 3]，颜色 #d4a855）从顶部贯穿至底部，覆盖整个Y轴。
- 竖线顶部显示事件名称标签（背景 #d4a855，白色文字，11px，圆角4px，最大120px截断）。
- 关键节点标注为纯展示，不可点击。不在当前可视范围时自动隐藏。

**Tooltip设计：**
- 鼠标在K线区域移动时显示十字光标，数据面板跟随鼠标。
- 面板内容：日期、开盘、收盘、最高、最低、涨跌额、涨跌幅。
- 涨跌额和涨跌幅正数红色、负数绿色。

**空状态：**
- Industry_asset为空 → 隐藏整个模块。
- 模拟数据生成异常 → 展示"数据加载异常" + 重试按钮（重新触发模拟数据生成）。

## 六、全局空状态与降级策略

| 场景 | 触发条件 | 处理策略 | 展示内容 |
|------|----------|----------|----------|
| 数据库零召回 | 所有关键词匹配策略均无结果 | 各模块均进入空状态 | 传播时间轴区域展示"未检索到相关文章，请调整关键词或时间区间" |
| 数据库连接异常 | UTS推送超时或报错 | 流水线中断 | 页面展示"数据加载失败" + 重试按钮 |
| 大模型观点提炼失败 | API超时/报错/格式不合法 | 首次失败自动重试1次，仍失败则观点列显示"提炼失败" | 观点总结模块保留文章基本信息 |
| 文章总数 < 3 | 区间内原创文章不足3篇 | 卡片墙按实际数量展示 | 热点文章卡片墙剩余位置留空 |
| key_points为空 | 意图识别未提取到关键节点 | 时间轴和K线图均不展示关键节点竖线 | 正常渲染其余内容 |
| Industry_asset为空 | 意图识别未提取到关联标的 | 隐藏关联标的走势模块 | 页面其余模块正常展示 |
| Embedding 服务异常 | 3秒内未返回或维度异常>30% | 跳过 Embedding，全部文章直接进入 LLM 精筛 | 正常渲染，无额外提示 |
| Embedding 相似度整体偏低 | 所有文章 score < 0.35 | 取 Top 50 进入 LLM 精筛，不再用阈值过滤 | 正常渲染，无额外提示 |
| LLM 精筛 API 异常 | 单批超时/报错，重试2次仍失败 | 该批全部保留（视为相关），其余批次正常精筛 | 正常渲染，记录日志 |
| LLM 精筛后文章极少 | 全部批次合并后 < 5篇 | 从 Embedding 结果按 score 取 Top 10 自动补回 | 正常渲染，无额外提示 |
| LLM 精筛后零文章 | 全部被判为不相关 | 取 Embedding Top 20 作为最终结果 | UI 提示"已展示语义最接近的文章" |
